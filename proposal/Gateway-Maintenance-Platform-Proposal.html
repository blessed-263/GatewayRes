Executive Summary

A working prototype, ready to scale.

We've already designed and built a functioning maintenance portal for Gateway: logins for supervisors, technicians and residents, mobile-first job management, photo evidence uploads, budgets and reporting. This proposal covers taking that prototype into a production system: hosted on your own subdomain, extended with native Android and Apple apps, and backed by one secure, shared database.

The platform is designed from day one to expand beyond maintenance, into a broader student services hub covering announcements, bookings, and self-service requests.

What Gateway gets

  • A branded web app on portal.gatewayres.co.za (or your preferred subdomain)
  • Android & iOS apps, optionally published on Google Play & the App Store
  • One secure, encrypted database shared across all platforms, no duplicate data, ever out of sync
  • Role-based access for supervisors, technicians and residents
  • A platform built to expand to full student self-service

The Opportunity

Maintenance requests shouldn't live in group chats and spreadsheets.

1

Slow, untracked reporting

Residents report issues informally, requests get lost, and there's no record of what was fixed, when, or by whom.

2

No visibility for management

Supervisors can't see workload, spend, or recurring problem areas across buildings without manually chasing technicians.

3

Fragmented, one-off tools

Off-the-shelf CMMS tools are generic, hard to white-label, and charge per-seat, with no path to a branded student-facing product.

Architecture & Security

One secure database. Three clients. No data silos.

All three apps talk to the same secured API layer, which is the only thing that ever touches the database. This means a resident's request, a technician's status update, and a supervisor's report are always looking at the same, current truth.

Notifications, built in

Every status change can trigger a notification automatically, no extra work for staff. Transactional emails are sent via Resend (new request confirmations, job assigned, job completed, reminders), the web and mobile apps show real-time in-app pop-ups/toasts the moment something changes, and the mobile apps add native push notifications on top, all from the same event, fired once from the API layer.

Security measures

  • Encryption in transit (TLS) and at rest for the database and file attachments
  • Role-based access control: residents, technicians and supervisors only ever see what they're permitted to
  • Signed, short-lived session tokens; passwords hashed, never stored in plain text
  • Full activity log for every request, status change and comment (auditability)

Reliability & operations

  • Automated daily database backups with point-in-time recovery
  • Hosted on a managed cloud provider (e.g. AWS / Azure) with regional failover
  • Staging environment for safe testing of new features before release
  • Monitoring & alerting for uptime, errors and unusual activity

Publishing to the App Stores

If Gateway wants the apps publicly listed, here's what's involved.

Google Play

Requires a one-time Google Play Developer account (~$25) registered to Gateway. Amalgamate prepares the listing, screenshots, privacy policy and handles submission. Typical review time: a few hours to 2 days.

Apple App Store

Requires an Apple Developer Program membership (~$99/year) under Gateway's organisation. Apple's review is stricter: typically 1-3 days, occasionally longer for first submissions. Amalgamate manages certificates, provisioning and the full submission process.

Note: publishing publicly is optional. Gateway can instead distribute the apps privately (e.g. via TestFlight for iOS or a direct APK / internal Play track for Android) to staff and residents only, skipping public store review entirely, useful for a phased rollout before going public.
